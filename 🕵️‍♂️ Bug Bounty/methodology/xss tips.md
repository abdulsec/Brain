Don't report Self-XSS. If your XSS attack requires a victim to do more than 1) click a link 2) navigate to a page or 3) perform a minimal page interaction (mouseover, single click), then don't report it. Always attempt to escalate via CSRF, javascript sink, or parameter reflection. 4. My first N/A report was a Self-Stored XSS. It really forced me to consider how important is to consider impact when writing up a report. I was able to get it triaged by crafting a Login CSRF that would land a victim a page with a stored HTML form that could be used to steal credentials. 5. Akamai WAF bypasses can sometimes be found on twitter, I found the `<marquee>` payload by just searching on twitter! new events and payloads to trick WAFs are discovered frequently, always be on the look out. 7. Another way to esclate self-stored XSS is store a payload is through a "backwards" IDOR or a session fixation attack. Essentially, if you have a page that's protected via some identifier (account number, hash, GUID, etc) that is 1) vulnerable to stored XSS (say, firstname) and 2) is publicly accessible, you can achieve the impact of a reflected XSS by storing a payload and sending the URL with the affixed identifier, causing XSS to execute in the context of the victim's session after clicking it. 8. Look for multiple reflections to help craft WAF and Filter bypasses. If two parameters have adjacent or close reflection contexts, you can spread out your payload into two separate payloads. The waf/filter doesn't have the context in mind when checking each payload independently, which can be abused to craft working payloads. In the example below, I used the first and last name fields (which were relfected next to each other with a space) to craft a working payload. 9. Some programs will accept the same vulnerability on multiple domains. It doesn't work every time because a program reserves the right to make claims of what constitutes a "shared root cause". However, if you can find a domain that mirrors one where you have already found an XSS, you can report it again. Another strategy is to wait for your original XSS to be fixed, then you have a better chance of having a second report triaged and rewarded because the vulns didn't technically "share" a root cause. 16. If you find a DOM-based open redirect, always attempt to escalate to XSS with javascript:alert(1). this is specific to DOM-based open redirect (i.e. `document.location="javascript:alert(1)"`) because a 302 with a redirect location header of "javascript:alert(document.domain)" or "data:text/html;base64,PHNjcmlwdD5hbGVydChkb2N1bWVudC5kb21haW4pOzwvc2NyaXB0Pg==" will not execute in modern browsers or the context of the targeted site. This can at most be considered HTML injection and is typically the same severity and payout as open redirect. I only recommend pursuing this if open redirect is explicitly not accepting by your target. 18.
